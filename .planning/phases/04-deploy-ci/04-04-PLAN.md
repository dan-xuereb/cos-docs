Create `/home/btc/github/cos-docs/.github/workflows/build.yml` following RESEARCH §8 (lines 439-573) with refinements below. Full required structure:

**Header block:**
```
name: Build and push docs image
```

**Triggers (on:):**
- `push: { branches: [main] }` — CI-02.
- `schedule: [{cron: '0 7 * * *'}]` — CI-01 + D-11 (07:00 UTC = 03:00 US-Eastern).
- `workflow_dispatch:` with one input:
  - `allow_partial` — description "Allow build to succeed with partial sibling failures", type `boolean`, default `false`. (Per RESEARCH Pitfall 12: even though typed boolean, the value arrives as string "true"/"false" in shell — compare with `[ "$X" = "true" ]`.)

**Concurrency (RESEARCH §8 + Pitfall 6):**
```
concurrency:
  group: cos-docs-build
  cancel-in-progress: false
```
Prevents nightly+manual race over `docs/api.md` swap/restore.

**Job:**
```
jobs:
  build:
    runs-on: self-hosted
    timeout-minutes: 45
    defaults:
      run:
        working-directory: /home/btc/github/cos-docs
```

**Steps (in order):**

1. **Sync cos-docs** — NO `actions/checkout` (RESEARCH §3 + Pitfall 1). In-place sync:
   ```
   - name: Sync cos-docs to target ref
     run: |
       git fetch origin
       git reset --hard "${GITHUB_SHA}"
       git clean -fdx -e site/ -e .venv-docs/ -e .venv-aggr/
   ```
   Safety: `-e site/` and `-e .venv-docs/` preserve pre-render venvs + outputs per RESEARCH safety note.

2. **Pull sibling repos + emit site-manifest.json** (D-02; continue-on-failure):
   ```
   - name: Pull sibling repos and emit manifest
     run: |
       set +e
       for dir in /home/btc/github/*/; do
         name=$(basename "$dir")
         [ "$name" = "cos-docs" ] && continue
         [ -d "$dir/.git" ] || continue
         echo "=== $name ==="
         git -C "$dir" pull --ff-only || echo "WARN: pull failed for $name"
       done
       set -e
       scripts/emit-site-manifest.sh > site-manifest.json
       jq -e '.repos | length >= 25' site-manifest.json
   ```

3. **Build per-repo API pages (--keep leaves swap in for aggregator):**
   ```
   - name: Build per-repo API pages
     run: scripts/build-all-api.sh --keep
   ```

4. **Build aggregator site:**
   ```
   - name: Build aggregator site
     run: |
       python3 -m venv .venv-aggr
       .venv-aggr/bin/pip install --quiet -r requirements-docs.txt
       .venv-aggr/bin/mkdocs build --strict
       mv site-manifest.json site/site-manifest.json
   ```

5. **Enforce strict-fail policy (D-04 + RESEARCH Pitfall 12 for boolean input):**
   ```
   - name: Enforce strict-fail policy
     env:
       ALLOW_PARTIAL: ${{ inputs.allow_partial }}
     run: |
       if [ "${{ github.event_name }}" != "workflow_dispatch" ]; then
         ALLOW_PARTIAL=false
       fi
       if grep -q "FAIL" .build-all-api-status.md; then
         echo "::warning::Partial sibling failures detected"
         cat .build-all-api-status.md
         if [ "$ALLOW_PARTIAL" != "true" ]; then
           echo "::error::Strict-fail: partial failure not permitted for ${{ github.event_name }}"
           exit 1
         fi
       fi
   ```

6. **Restore sibling docs/api.md (always, even on failure) — RESEARCH §9 Pitfall 4:**
   ```
   - name: Restore per-repo docs/api.md
     if: always()
     run: scripts/build-all-api.sh --restore
   ```

7. **Upload status artifact on failure:**
   ```
   - name: Upload build-all-api-status on failure
     if: failure()
     uses: actions/upload-artifact@v4
     with:
       name: build-all-api-status
       path: /home/btc/github/cos-docs/.build-all-api-status.md
       if-no-files-found: ignore
   ```

8. **Build + push image (RESEARCH §2; D-03 + D-09: :latest on every green build, :nightly-YYYYMMDD only on schedule):**
   ```
   - name: Build and push image
     run: |
       SHORT_SHA="${GITHUB_SHA::7}"
       SHA_TAG="10.70.0.30:5000/cos-docs:${SHORT_SHA}"
       LATEST_TAG="10.70.0.30:5000/cos-docs:latest"
       docker build -t "$SHA_TAG" -t "$LATEST_TAG" .
       docker push "$SHA_TAG"
       docker push "$LATEST_TAG"
       if [ "${{ github.event_name }}" = "schedule" ]; then
         NIGHTLY_TAG="10.70.0.30:5000/cos-docs:nightly-$(date -u +%Y%m%d)"
         docker tag "$SHA_TAG" "$NIGHTLY_TAG"
         docker push "$NIGHTLY_TAG"
         echo "::notice::Nightly tag pushed: $NIGHTLY_TAG"
       fi
       echo "::notice::Image pushed: $SHA_TAG + $LATEST_TAG"
   ```

9. **Deploy hint (D-06 — final log line):**
   ```
   - name: Deploy hint
     run: |
       echo "→ deploy with: kubectl rollout restart deployment/cos-docs -n cos-docs"
   ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub workflow runtime → runner host FS | Workflow runs bash as user btc; has full read/write over /home/btc/github/* and /var/run/docker.sock. |
| Runner → registry | anonymous docker push to 10.70.0.30:5000 over plaintext HTTP (internal VLAN). |
| main branch writes → workflow trigger | Any push to main triggers a build; limit contributors with GH branch protection if needed (out of scope here). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04-01 | Tampering | sibling docs/api.md swap never restored | mitigate | `if: always()` on restore step; RESEARCH Pitfall 4. |
| T-04-04-02 | Denial of Service | concurrent nightly + dispatch runs race on swap/mv | mitigate | `concurrency.group: cos-docs-build` + `cancel-in-progress: false` serializes runs. |
| T-04-04-03 | Information Disclosure | build logs leak env/secrets | accept | No secrets used in this workflow (anonymous registry); logs only contain public commit SHAs + file names. |
| T-04-04-04 | Elevation of Privilege | workflow_dispatch allow_partial misuse | mitigate | allow_partial only bypasses strict-fail for operator-triggered dispatch runs; schedule + push always strict. |
| T-04-04-05 | Spoofing | boolean input arrives as string (RESEARCH Pitfall 12) | mitigate | Comparison pattern `[ "$ALLOW_PARTIAL" = "true" ]` handles both forms; schedule/push defensively override to false. |
</threat_model>
